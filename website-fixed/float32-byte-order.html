Modbus Float32 Byte Order & "Wrong Values"

You're reading a voltage register and getting 3.87√ó10¬≤‚Å¥ instead of 236V. The data is there ‚Äî it's just being interpreted in the wrong byte order. Here's how to fix it every time.

Data Types Updated Feb 2026 9 min read

Why This Happens

Modbus registers are 16 bits (2 bytes) each. A Float32 (IEEE 754 single-precision floating point) needs 32 bits ‚Äî so it's split across two consecutive registers. The problem is that the Modbus specification doesn't define which register holds the high word and which holds the low word.

Each register also contains 2 bytes, and while Modbus defines big-endian byte order within a register, the word order (which register comes first) is left to the manufacturer. The result: four possible interpretations of the same two registers.

The Four Byte Orders

Let's say the float value is 236.38 (a typical UK mains voltage). In IEEE 754, this is hex 43 6C 61 48, which we label as bytes A B C D:

üî• The Dangerous Case

Some wrong byte orders produce values that look plausible but are wrong. If you're reading a temperature and get 25.4¬∞C when the actual value is 42.5¬∞C, the values are in the same ballpark and you might not notice the error for weeks. Always verify against a known reference.

How to Identify the Correct Byte Order

Method 1: Read the Documentation

The device's Modbus protocol manual should state the byte order. Look for terms like:

  • "Big-endian" / "MSB first" / "High word first" ‚Üí AB CD
  • "Low word first" / "Little-endian word order" ‚Üí CD AB
  • "Modicon compatible" / "Modicon standard" ‚Üí CD AB (Modicon PLCs used low-word-first)

Method 2: Use a Known Value

Read the raw register values (as UINT16) for a measurement you can independently verify, then manually decode all four byte orders:

  1. Read register N and N+1 as unsigned integers
  2. Convert each to hex (e.g., register N = 17260 ‚Üí 0x436C)
  3. Try all four combinations and see which produces the expected value

Method 3: Look for the Exponent

A quick diagnostic trick: in IEEE 754, the exponent is in bits 23-30, which means it's in the most significant byte. For typical industrial values (voltages, currents, temperatures), the MSB is usually in the range 0x40-0x47. Whichever byte order puts that byte first is probably correct.

Handling Byte Order in PLC Code

PLCnext Structured Text (IEC 61131-3)

// If your device sends CD AB (word-swapped) and your PLC expects AB CD:
// Swap the two 16-bit words before interpreting as REAL

VAR
    wRawHigh : WORD;    // Register N   (contains low word CD)
    wRawLow  : WORD;    // Register N+1 (contains high word AB)
    dwCombined : DWORD;
    rResult  : REAL;
END_VAR

// Swap words: put high word in MSW, low word in LSW
dwCombined := SHL(WORD_TO_DWORD(wRawLow), 16) OR WORD_TO_DWORD(wRawHigh);

// Interpret the DWORD bits as REAL (type-pun)
rResult := DWORD_TO_REAL(dwCombined);

Python (pymodbus)

from pymodbus.payload import BinaryPayloadDecoder
from pymodbus.constants import Endian

# Read 2 registers
result = client.read_holding_registers(address=2999, count=2, slave=1)

# Decode as Float32 ‚Äî try different byte orders if wrong
decoder = BinaryPayloadDecoder.fromRegisters(
    result.registers,
    byteorder=Endian.BIG,    # Byte order within register
    wordorder=Endian.BIG     # Word order across registers
)
voltage = decoder.decode_32bit_float()
print(f"Voltage: {voltage:.2f} V")
