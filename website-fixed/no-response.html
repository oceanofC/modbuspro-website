Modbus "No Response from Slave" ‚Äî Complete Fix Guide

The most common Modbus failure. Your master sends a request, but nothing comes back. Here's a systematic approach to diagnose and fix it in minutes, not hours.

Troubleshooting Updated Feb 2026 8 min read

Why This Happens

A "no response" means your Modbus master sent a valid request frame, but received absolutely nothing back within the timeout period. This is distinct from an exception response (where the slave replies with an error code) ‚Äî here, the slave is either not receiving your request, or not able to reply.

Quick Checklist (Fix 80% of Cases)

Before going deep, run through these in order. Most "no response" issues are caught in the first five checks:

  1. Is the slave powered on? ‚Äî Yes, this happens. Especially during commissioning when circuits are being energised in phases.
  2. Is Modbus enabled on the slave? ‚Äî Many devices ship with Modbus communication disabled by default. Check the device settings/menu.
  3. Does the slave address match? ‚Äî Verify the Unit ID / slave address configured on the device matches what your master is polling. Default is often 1, but not always.
  4. Are comm parameters identical? ‚Äî For RTU: baud rate, parity, data bits, and stop bits must be exactly the same on both sides.
  5. Is the cable connected and correct? ‚Äî For RS-485: check A/B polarity. For TCP: can you ping the IP address?

Modbus RTU (Serial) Diagnostics

1. Verify Wiring

RS-485 is a differential signal using two wires. The naming convention is a mess across manufacturers:

The safest approach: if you get no response, swap A and B. It won't damage anything and it's the most common wiring mistake.

2. Check Termination Resistors

RS-485 networks need 120Œ© termination resistors at both ends of the bus (not at every device). Missing termination causes signal reflections that corrupt frames ‚Äî the slave receives garbage and doesn't respond.

3. Ground Reference

Despite being called "2-wire," RS-485 needs a common ground reference between master and slave. Without it, the common-mode voltage can exceed the receiver's range, especially over long cable runs or between different power sources.

4. Baud Rate & Framing

Every parameter must match exactly:

Master:  9600 baud, 8 data bits, Even parity, 1 stop bit
Slave:   9600 baud, 8 data bits, Even parity, 1 stop bit
         ‚Üë ALL must be identical
‚ö† The Parity/Stop Bit Trap

The Modbus spec says: if parity is None, use 2 stop bits. But many devices don't support 2 stop bits and just use 1. If you've set "None" parity with 1 stop bit on one side and 2 on the other, you'll get intermittent failures or complete silence. Switch to Even parity, 1 stop bit to avoid this entirely.

Modbus TCP Diagnostics

1. Network Layer First

# Can you reach the device?
ping 192.168.1.100

# Is port 502 open?
telnet 192.168.1.100 502

# Or use netcat
nc -zv 192.168.1.100 502

If ping fails, the problem is IP/subnet/VLAN configuration ‚Äî not Modbus. If ping works but port 502 is refused, the device's Modbus service may be disabled or a firewall is blocking it.

2. Unit ID for TCP

Modbus TCP still uses a Unit Identifier (like slave address). For direct device connections, try Unit ID 1 or 0 (some devices accept either). For gateways that bridge TCP to RTU, the Unit ID maps to the downstream RS-485 slave address.

3. Connection Limits

Many industrial devices only allow 2-5 simultaneous TCP connections. If your SCADA is connected, an HMI is connected, and then you try a diagnostic tool ‚Äî the device may reject the new connection silently. Close other connections or check the device's max connection setting.

Still No Response? Advanced Diagnostics

Use a Protocol Analyser

For RTU: use a second USB-RS485 adapter in listen-only mode to see what's actually on the bus. You should see your master's request frames. If you don't, the problem is between your master and the bus. If you see requests but no replies, the slave is the issue.

For TCP: use Wireshark with a Modbus TCP dissector. Filter on tcp.port == 502 and check whether responses are being sent.

Check the Slave's Diagnostic Counters

Many devices maintain internal communication counters accessible through their local HMI or web interface. Check for:

  • Messages received ‚Äî if this is incrementing, the slave IS receiving your requests (problem is in the reply path)
  • CRC errors ‚Äî the slave is receiving corrupted data (wiring/EMI issue)
  • Exception responses sent ‚Äî the slave is replying, but something is eating the response on the way back
